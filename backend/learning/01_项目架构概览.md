## 1.2 架构设计思路

### 1.2.1 为什么采用微服务架构？

#### 设计决策：服务边界划分

**思考过程：**

```
问题：我们需要一个RAG智能客服系统，应该如何划分服务？

分析：
┌────────────────────────────────────────────────────┐
│ 需求清单                                            │
├────────────────────────────────────────────────────┤
│ 1. 用户认证和管理                                  │
│ 2. 企业管理和多租户                                │
│ 3. 聊天对话和RAG检索                              │
│ 4. 知识库文档管理                                 │
│ 5. 工具调用（订单查询等）                        │
│ 6. 多渠道接入（电商平台等）                      │
│ 7. 系统监控和告警                                 │
│ 8. 数据分析                                       │
└────────────────────────────────────────────────────┘

决策：
┌────────────────────────────────────────────────────┐
│ 服务拆分原则                                        │
├────────────────────────────────────────────────────┤
│ 1. 单一职责：每个服务只做一件事                    │
│ 2. 独立部署：可以独立扩展和升级                  │
│ 3. 技术异构：不同服务可用不同技术栈              │
│ 4. 故障隔离：一个服务挂了不影响其他服务          │
└────────────────────────────────────────────────────┘
```

**最终划分：**

| 服务 | 职责 | 边界划分理由 |
|------|------|-------------|
| User Service | 用户+企业+认证 | 这些都围绕"账户"展开 |
| Chat Service | 对话+RAG+Agent | 核心业务逻辑，必须在一起 |
| Knowledge Service | 文档+向量 | 数据密集型，独立优化 |
| Channel Service | 渠道接入 | 面向外部系统的适配层 |
| Alert Service | 告警+人工 | 事件驱动型，独立事件流 |
| Analytics Service | 统计+报表 | 读密集型，独立优化 |
| Admin Service | 系统管理 | 管理功能，权限特殊 |
| API Gateway | 网关功能 | 横切关注点，统一处理 |

#### 设计权衡

**优点：**
- ✅ 独立扩展：根据负载独立扩容
- ✅ 技术灵活：不同服务用最合适的技术
- ✅ 故障隔离：一个问题不会拖垮整个系统
- ✅ 团队自治：不同团队负责不同服务

**缺点：**
- ❌ 复杂度高：服务间通信、分布式事务
- ❌ 运维复杂：需要服务发现、监控、日志聚合
- ❌ 延迟增加：跨服务调用比本地调用慢

**决策**：接受这些权衡，因为系统需要支持多渠道接入、可能需要独立扩展某个服务、团队可能并行开发。

### 1.2.2 为什么采用前后端分离？

#### 设计决策：前端应用划分

**思考过程：**

```
问题：前端应该是一个应用还是多个？如何组织？

分析：
┌────────────────────────────────────────────────────┐
│ 用户角色                                            │
├────────────────────────────────────────────────────┤
│ 1. 终端用户：使用客服对话功能                      │
│ 2. 企业管理员：管理知识库、查看统计               │
│ 3. 系统管理员：管理整个平台                       │
└────────────────────────────────────────────────────┘

方案A：一个应用，通过权限控制显示不同功能
┌─────────┐
│  单一应用  │  → 缺点：安全风险高，权限控制复杂
└─────────┘

方案B：三个独立应用
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 用户端   │  │ 企业端   │  │ 系统端   │  → 优点：职责分离，安全隔离
└─────────┘  └─────────┘  └─────────┘
```

**决策**：采用方案B，三个独立应用

**理由：**
1. **安全隔离**：系统管理员和企业管理员的功能必须严格分离
2. **职责分离**：不同角色的用户界面差异很大
3. **独立部署**：可以独立升级，不互相影响
4. **团队分工**：不同团队可以负责不同应用

#### Monorepo决策

**为什么前端用Monorepo？**

```
问题：三个应用是分开仓库还是放在一个仓库？

方案A：分开仓库
├─ consumer-app/
├─ enterprise-app/
└─ admin-app/

方案B：Monorepo
└─ frontend/
    ├─ apps/
    │  ├─ consumer/
    │  ├─ enterprise/
    │  └─ admin/
    └─ packages/
       ├─ ui/
       ├─ api-client/
       └─ store/
```

**决策**：Monorepo

**理由：**
1. **代码共享**：三个应用共享UI组件、API客户端
2. **统一依赖**：避免版本不一致
3. **原子提交**：一次提交可以改多个应用
4. **统一工具链**：统一的构建、测试、lint

## 1.3 服务间通信机制

### 1.3.1 同步通信：REST API

**使用场景**：需要实时响应的操作

```
┌──────────────────────────────────────────────────────┐
│              REST API 通信模式                       │
├──────────────────────────────────────────────────────┤
│                                                       │
│   Client                                               │
│      │                                                 │
│      │  POST /api/v1/chat/sessions                   │
│      ↓                                                 │
│   API Gateway                                         │
│      │                                                 │
│      │  路由到 Chat Service                           │
│      ↓                                                 │
│   Chat Service                                        │
│      │                                                 │
│      │  调用 Knowledge Service                        │
│      ↓                                                 │
│   Knowledge Service                                   │
│      │                                                 │
│      │  返回结果                                      │
│      ↓                                                 │
│   Chat Service → API Gateway → Client                │
│                                                       │
└──────────────────────────────────────────────────────┘
```

**设计要点：**

```python
# API Gateway 中的路由配置
# 文件：backend/api-gateway/main.py

@app.post("/api/v1/chat/sessions")
async def create_session(request: Request):
    """创建聊天会话"""
    
    # 1. 验证请求
    body = await request.json()
    validate_session_request(body)
    
    # 2. 调用后端服务
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{CHAT_SERVICE_URL}/sessions",
            json=body,
            headers={"X-Enterprise-ID": enterprise_id}
        )
    
    # 3. 返回结果
    return response.json()
```

**为什么不用gRPC？**

- 理由1：HTTP/REST更容易调试和集成
- 理由2：前端可以直接调用（通过API Gateway）
- 理由3：项目规模还不需要gRPC的性能

### 1.3.2 服务注册与发现

**设计决策：为什么不使用服务注册中心（如Consul/Nacos）？**

```
问题：服务之间如何知道彼此的地址？
```

### 1.3.3 为什么不使用消息队列？

**思考：什么时候需要消息队列？**

```
需要消息队列的场景：
1. 异步任务：不需要立即返回结果
2. 削峰填谷：处理突发流量
3. 事件驱动：多个服务响应同一个事件
4. 可靠投递：确保消息不丢失

本项目的场景：
1. 大部分请求需要同步返回（对话需要实时响应）
2. 流量相对平稳
3. 没有复杂的事件驱动需求
```

**决策**：暂不使用消息队列

**理由：**
1. 当前场景主要是同步请求
2. Redis已经可以满足临时消息存储需求
3. 保持架构简单

**未来扩展**：如果需要支持人工转接的即时通知、批量文档处理等，可以引入Celery或Kafka。

### 1.4.2 多租户隔离策略

**设计决策：隔离级别选择**

| 隔离级别 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 共享数据库+共享Schema | 成本最低 | 隔离性差 | 隔离要求低 |
| 共享数据库+独立Schema | 平衡成本和隔离 | 运维复杂 | 中等隔离 |
| 独立数据库 | 隔离性最好 | 成本高 | 高隔离要求 |

**决策**：共享数据库+企业ID字段隔离

**理由：**
1. **成本**：不需要为每个企业创建独立数据库
2. **运维**：数据库数量可控
3. **隔离性**：通过enterprise_id字段保证逻辑隔离
4. **灵活性**：可以随时迁移某个企业到独立数据库

**实现代码：**

```python
# 每个数据库查询都强制带上 enterprise_id
# 文件：backend/common/security.py

def get_enterprise_id_from_token(token: str) -> UUID:
    """
    从JWT Token中解析企业ID
    
    Token结构：
    {
        "sub": "user_id",
        "enterprise_id": "uuid",
        "role": "admin",
        "exp": 1234567890
    }
    """
    payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    return UUID(payload["enterprise_id"])


# 在所有查询中注入 enterprise_id
class EnterpriseFilter:
    """企业数据过滤"""
    
    @staticmethod
    def filter_by_enterprise(session, model_class, enterprise_id: UUID):
        """
        自动添加企业ID过滤
        
        使用示例：
        sessions = EnterpriseFilter.filter_by_enterprise(
            db, ChatSession, enterprise_id
        )
        """
        return session.query(model_class).filter(
            model_class.enterprise_id == enterprise_id
        )
```

**RBAC实现：**

```python
class Role:

    ENTERPRISE_ADMIN = "enterprise_admin"
    AGENT = "agent"

class Permission:
    """权限定义"""
    # 知识库权限
    KB_CREATE = "kb:create"
    KB_READ = "kb:read"
    KB_UPDATE = "kb:update"
    KB_DELETE = "kb:delete"
    
    # 用户权限
    USER_CREATE = "user:create"
    USER_READ = "user:read"
    USER_MANAGE = "user:manage"  # 只能管理同企业的用户
    
    # 系统权限
    SYSTEM_VIEW = "system:view"
    SYSTEM_MANAGE = "system:manage"


# 角色-权限映射
ROLE_PERMISSIONS = {
    Role.PLATFORM_ADMIN: [
        Permission.KB_CREATE, Permission.KB_READ, Permission.KB_UPDATE, Permission.KB_DELETE,
        Permission.USER_CREATE, Permission.USER_READ, Permission.USER_MANAGE,
        Permission.SYSTEM_VIEW, Permission.SYSTEM_MANAGE
    ],
    Role.ENTERPRISE_ADMIN: [
        Permission.KB_CREATE, Permission.KB_READ, Permission.KB_UPDATE, Permission.KB_DELETE,
        Permission.USER_CREATE, Permission.USER_READ, Permission.USER_MANAGE,
        Permission.SYSTEM_VIEW
    ],
    Role.AGENT: [
        Permission.KB_READ,
        Permission.USER_READ
    ],
    Role.USER: [
        Permission.KB_READ,
        Permission.USER_READ  # 只读自己
    ]
}


def check_permission(role: str, permission: str) -> bool:
    """检查角色是否有指定权限"""
    return permission in ROLE_PERMISSIONS.get(role, [])
```

```python
# backend/api-gateway/security.py

class JWTAuth:
    """JWT认证工具"""
    
    SECRET_KEY = "your-secret-key-change-in-production"
    ALGORITHM = "HS256"
    
    @classmethod
    def create_token(cls, user_id: str, enterprise_id: str, role: str) -> str:
        """
        创建JWT Token
        
        Token中包含：
        - sub: 用户ID
        - enterprise_id: 企业ID（用于多租户）
        - role: 用户角色
        - exp: 过期时间
        """
        expire = datetime.utcnow() + timedelta(days=7)
        
        payload = {
            "sub": user_id,
            "enterprise_id": enterprise_id,
            "role": role,
            "exp": expire
        }
        
        token = jwt.encode(payload, cls.SECRET_KEY, algorithm=cls.ALGORITHM)
        return token
    
    @classmethod
    def verify_token(cls, token: str) -> Optional[dict]:
        """验证Token并返回Payload"""
        try:
            payload = jwt.decode(token, cls.SECRET_KEY, algorithms=[cls.ALGORITHM])
            return payload
        except jwt.ExpiredSignatureError:
            raise AuthenticationError("Token已过期")
        except jwt.InvalidTokenError:
            raise AuthenticationError("无效的Token")
    
    @classmethod
    def extract_enterprise_id(cls, token: str) -> str:
        """从Token中提取企业ID"""
        payload = cls.verify_token(token)
        return payload["enterprise_id"]
```

```python
# backend/chat-service/sse_stream.py

class SSEStreamer:
    """SSE流式响应工具"""
    
    @staticmethod
    def event(event_type: str, data: dict, event_id: Optional[int] = None) -> str:
        """
        生成SSE格式的事件
        
        Args:
            event_type: 事件类型 (message, error, done)
            data: 事件数据
            event_id: 事件ID（可选）
        
        Returns:
            SSE格式的字符串
        """
        lines = [
            f"event: {event_type}",
            f"data: {json.dumps(data, ensure_ascii=False)}"
        ]
        
        if event_id:
            lines.append(f"id: {event_id}")
        
        lines.append("")  # 空行结束事件
        return "\n".join(lines) + "\n"
```
