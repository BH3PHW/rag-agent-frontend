## 11.1 多租户架构概述

### 11.1.1 为什么需要多租户？

**产品化思维**：企业级SaaS的核心需求

```
传统单租户问题：
❌ 每个企业独立部署，成本高
❌ 管理复杂，升级困难
❌ 数据孤岛，无法跨企业分析
❌ 资源利用率低

多租户解决方案：
✅ 共享基础设施，成本分摊
✅ 统一管理，版本同步
✅ 数据集中，便于分析
✅ 资源池化，高利用率
```

### 11.1.2 多租户隔离级别

| 隔离级别 | 说明 | 成本 | 适用场景 |
|---------|------|------|---------|
| **数据库隔离** | 每个租户独立数据库 | 高 | 金融、医疗等高安全场景 |
| **Schema隔离** | 同一数据库，不同Schema | 中 | 中等安全要求 |
| **行级隔离** | 同一表，通过租户ID区分 | 低 | 大多数SaaS场景 |

**本项目采用：行级隔离 + 企业ID**

```python
def extract_enterprise_context(token: str) -> Dict[str, Any]:
    """从Token中提取企业上下文"""
    try:
        payload = verify_token(token)  # 验证JWT
        return {
            "user_id": payload.get("sub"),
            "enterprise_id": payload.get("enterprise_id"),
            "role": payload.get("role")
        }
    except JWTError:
        return None
```

## 11.4 企业隔离策略

### 11.4.1 API层隔离

**工程化实现**：请求级别的企业验证

```python
async def validate_enterprise_request(
    request: Request,
    enterprise_id: str
) -> str:
    """
    验证企业访问权限

    确保请求携带的Token属于目标企业

    产品化：防止跨企业数据访问
    """
    token = request.headers.get("Authorization")
    if not token:
        raise HTTPException(401, "Authentication required")

    # 从Token中提取企业ID
    token_enterprise_id = extract_enterprise_context(token)["enterprise_id"]

    # 验证企业ID匹配
    if token_enterprise_id != enterprise_id:
        raise HTTPException(403, "Access denied: enterprise mismatch")

    return enterprise_id
```

### 11.4.2 数据库层隔离

```python
class EnterpriseScopedRepository:
    """企业范围限定的数据仓库"""

    def __init__(self, user_context: UserContext):
        self.user_context = user_context

    async def get_knowledge_bases(self):
        """获取知识库 - 自动过滤企业数据"""
        query = f"""
            SELECT * FROM knowledge_bases
            WHERE enterprise_id = '{self.user_context.enterprise_id}'
        """
        return await db.execute(query)

    async def create_knowledge_base(self, data: dict):
        """创建知识库 - 自动注入企业ID"""
        data["enterprise_id"] = self.user_context.enterprise_id
        return await db.insert("knowledge_bases", data)

# 使用示例
```

```python
def check_quota(enterprise_id: str, resource: str) -> bool:
    """检查企业配额"""
    enterprise = get_enterprise(enterprise_id)
    quota = PlanQuota.__dict__[enterprise.plan.upper()]

    current_usage = get_resource_usage(enterprise_id, resource)

    limit = quota[f"max_{resource}"]
    if limit == -1:  # 无限制
        return True

    return current_usage < limit
```

## 11.7 工程化最佳实践

### 11.7.1 中间件设计

```python
@app.middleware("http")
async def enterprise_context_middleware(request: Request, call_next):
    """企业上下文中间件 - 工程化：透明处理"""
    # 1. 提取Token
    token = request.headers.get("Authorization")

    # 2. 解析企业上下文
    context = extract_enterprise_context(token) if token else None

    # 3. 注入到请求状态
    request.state.enterprise_context = context

    # 4. 处理请求
    response = await call_next(request)

    # 5. 记录审计日志
    if context and request.url.path.startswith("/api/"):
        log_api_access(context, request)

    return response
```

### 11.7.2 依赖注入

```python
class EnterpriseContext:
    """企业上下文 - 工程化：统一注入"""

    def __init__(self, request: Request):
        self.context = request.state.enterprise_context

    @property
    def enterprise_id(self) -> str:
        return self.context["enterprise_id"] if self.context else None

    @property
    def user_id(self) -> str:
        return self.context["user_id"] if self.context else None

    @property
    def is_admin(self) -> bool:
        return self.context.get("role") == "admin" if self.context else False

# 使用依赖注入
@router.post("/enterprise/config")
async def update_enterprise_config(
    data: dict,
    ctx: EnterpriseContext = Depends()
):
    # 只需要一行，就能获取完整的企业上下文
    if not ctx.is_admin:
        raise PermissionDeniedError()

    return update_enterprise(ctx.enterprise_id, data)
```

## 11.9 本章小结

### 核心概念

| 概念 | 说明 |
|------|------|
| 多租户隔离 | 通过enterprise_id实现数据隔离 |
| JWT Token | 包含企业上下文 |
| 套餐配额 | 分层定价与资源控制 |
| 企业审计 | 完整的操作追溯 |

### 工程化思维要点

1. **数据隔离** - 企业ID贯穿所有数据操作
2. **统一验证** - 中间件统一处理企业上下文
3. **审计日志** - 完整的企业级操作追溯
4. **可扩展性** - 支持新增企业类型和套餐

### 产品化思维要点

1. **分层套餐** - Basic/Pro/Enterprise，满足不同需求
2. **配额管理** - 防止资源滥用，保障服务质量
3. **企业后台** - 数据可视化，管理更便捷
4. **安全告警** - 保护企业数据安全
