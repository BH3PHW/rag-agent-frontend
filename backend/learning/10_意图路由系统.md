## 10.1 为什么需要意图路由？

### 10.1.1 问题背景

在智能客服系统中，直接调用大模型的成本是很高的：

```
问题分析：
1. 成本高昂：每次调用大模型都需要花费 Token
2. 响应延迟：LLM 推理需要时间，简单问题等待太久
3. 资源浪费：寒暄、简单问题不需要 LLM 能力
4. 质量保证：特定业务场景需要精确处理，不能依赖 LLM
```

### 10.1.2 产品化思维：成本与体验的平衡

| 场景 | 传统方案 | 问题 | 路由方案 | 收益 |
|------|---------|------|---------|-----|
| 寒暄闲聊 | LLM 响应 | 浪费 Token | 规则回复 | 100% 成本节省 |
| 订单查询 | LLM + API | 响应慢 | 直接业务接口 | 响应快 10 倍 |
| 复杂咨询 | LLM + RAG | 成本高 | 完整流程 | 保证质量 |
| 投诉转人工 | LLM 处理 | 风险高 | 立即转人工 | 用户体验好 |

```python
class IntentType(Enum):
    """意图类型枚举（工程化：类型安全）"""
    GREETING = "greeting"           # 寒暄
    THANKS = "thanks"              # 感谢
    FAREWELL = "farewell"          # 告别
    ORDER_QUERY = "order_query"     # 订单查询
    PRODUCT_QUERY = "product_query" # 产品查询
    COMPLAINT = "complaint"        # 投诉
    FAQ = "faq"                    # FAQ匹配
    RAG = "rag"                    # RAG检索
    HUMAN = "human"                # 转人工
    UNKNOWN = "unknown"            # 未知问题

class CostTier(Enum):
    """成本等级（产品化：成本控制）"""
    FREE = 0        # 免费（规则响应）
    LOW = 1         # 低成本
    MEDIUM = 2      # 中等成本
    HIGH = 3        # 高成本

@dataclass
class IntentDecision:
    """意图路由决策（工程化：统一数据结构）"""
    intent: IntentType
    cost_tier: CostTier
    should_use_llm: bool
    should_use_rag: bool
    should_escalate: bool
    confidence: float
    response: Optional[str]
    extracted_entities: Dict[str, Any]
    reasoning: str
```

## 10.3 规则匹配层

### 10.3.1 正则表达式模式设计

```python
class IntentRule:
    """
    意图路由规则定义

    工程化思维：
    1. 使用正则表达式进行快速匹配（毫秒级）
    2. 模式分层设计，先匹配高确定性意图
    3. 可扩展设计，方便新增意图类型
    """

    GREETING_PATTERNS = [
        r"^(你好|您好|hi|hello|hey)$",
        r"^(在|在吗|在吗？|在不在)$",
        r"^(早上好|下午好|晚上好|早安|晚安)$",
    ]

    THANKS_PATTERNS = [
        r"^(谢谢|感谢|多谢|谢了|谢谢您)$",
        r"^(太感谢了|非常感谢|十分感谢)$",
    ]

    FAREWELL_PATTERNS = [
        r"^(再见|拜拜|拜|下次见|回头见|走了)$",
    ]

    ORDER_PATTERNS = [
        r"(订单|单号|快递|发货|物流)",
        r"(什么时候到|什么时候发货|发货了吗)",
        r"(查订单|看订单|我的订单)",
    ]

    # ... 更多模式定义

    @classmethod
    def match(cls, text: str) -> Optional[IntentType]:
        """根据文本内容匹配意图类型"""
        text_lower = text.lower().strip()

        # 工程化：按优先级匹配
        if cls._match_patterns(text_lower, cls.GREETING_PATTERNS):
            return IntentType.GREETING

        if cls._match_patterns(text_lower, cls.THANKS_PATTERNS):
            return IntentType.THANKS

        if cls._match_patterns(text_lower, cls.FAREWELL_PATTERNS):
            return IntentType.FAREWELL

        # ... 更多匹配逻辑

        return None
```

## 10.4 实体提取层

### 10.4.1 关键实体提取

```python
@classmethod
def extract_entities(cls, text: str, intent: IntentType) -> Dict[str, Any]:
    """根据意图类型提取关键实体（工程化：上下文相关）"""
    entities = {}

    if intent == IntentType.ORDER_QUERY:
        # 订单号提取（工程化：模式匹配）
        order_id_match = re.search(r"([A-Z0-9]{8,})", text)
        if order_id_match:
            entities["order_id"] = order_id_match.group(1)

        # 手机号提取
        phone_match = re.search(r"1[3-9]\d{9}", text)
        if phone_match:
            entities["phone"] = phone_match.group()

    elif intent == IntentType.PRODUCT_QUERY:
        # 产品型号提取
        product_patterns = [
            (r"(A\d+|X\d+|Pro|Max)", "model_series"),
            (r"(\d+)寸|(\d+)英寸", "size"),
            (r"(\d+)元|(\d+)块", "price_range"),
        ]
        for pattern, entity_type in product_patterns:
            match = re.search(pattern, text)
            if match:
                entities[entity_type] = match.group()

    return entities
```

### 10.4.2 工程化实践：预提取的价值

```
实体提取的好处：
1. 减少 LLM 调用：直接给业务接口，不需要 LLM 理解
2. 提高准确率：正则匹配比 LLM 更可靠
3. 加速响应：毫秒级提取 vs 秒级 LLM 理解
4. 降低成本：节省 Token
```

## 10.6 成本统计层

### 10.6.1 成本计算模型

```python
# Token 成本估算（基于 Qwen Turbo）
COST_PER_TOKEN = 0.002  # 每 Token 成本（元）
AVG_QUERY_TOKENS = 500   # 平均查询 Token 数
AVG_RESPONSE_TOKENS = 300 # 平均响应 Token 数

def calculate_cost_saved(self, decision: IntentDecision) -> float:
    """
    计算成本节省（产品化思维：ROI 追踪）

    如果不经过意图路由直接调用 LLM 的成本
    减去当前决策的实际成本
    """
    full_llm_cost = (self.AVG_QUERY_TOKENS + self.AVG_RESPONSE_TOKENS) * self.COST_PER_TOKEN

    if decision.cost_tier == CostTier.FREE:
        return full_llm_cost  # 100% 节省
    elif decision.cost_tier == CostTier.LOW:
        return full_llm_cost * 0.3  # 节省 70%
    elif decision.cost_tier == CostTier.MEDIUM:
        return full_llm_cost * 0.5  # 节省 50%
    else:
        return 0.0  # 无节省
```

### 10.6.2 日志与统计

```python
def get_cost_stats(self) -> Dict[str, Any]:
    """获取成本统计（工程化：可观测性）"""
    if not self.route_logs:
        return {
            "total_requests": 0,
            "total_tokens_saved": 0,
            "total_cost_saved": 0,
            "llm_call_rate": 0
        }

    total = len(self.route_logs)
    llm_calls = sum(1 for log in self.route_logs if log.llm_called)
    total_saved = sum(log.estimated_cost_saved for log in self.route_logs)

    return {
        "total_requests": total,
        "total_tokens_saved": int(total_saved / self.COST_PER_TOKEN),
        "total_cost_saved": round(total_saved, 4),
        "llm_call_rate": round(llm_calls / total * 100, 2),
        "intent_distribution": self._get_intent_distribution()
    }
```

## 10.7 工程化实践

### 10.7.1 集成到 FastAPI

```python
# 在 FastAPI 中使用意图路由
from fastapi import FastAPI, Depends, Request

app = FastAPI()

@app.middleware("http")
async def intent_routing_middleware(request: Request, call_next):
    """
    工程化：中间件集成

    优势：
    1. 透明处理，业务代码无需改动
    2. 统一入口，便于监控
    3. 可以轻松禁用/启用
    """
    # 只有特定路由才走意图路由
    if request.url.path.startswith("/api/chat"):
        # 获取请求体
        body = await request.json()
        query = body.get("query", "")

        # 意图分类
        router = get_intent_router()
        decision = router.classify(query)

        # 记录决策到请求上下文
        request.state.intent_decision = decision

        # 如果有预设回复，直接返回
        if decision.response and not decision.should_use_llm:
            return {
                "response": decision.response,
                "intent": decision.intent.value,
                "routed": True
            }

    # 继续正常处理
    response = await call_next(request)
    return response
```

## 10.8 产品化思维

### 10.8.1 A/B 测试框架

```python
"""
产品化：不同路由策略的 A/B 测试

可以对比：
1. 规则路由 vs 完全 LLM
2. 不同的置信度阈值
3. 不同的回复库
4. 新意图类型的效果
"""

class ABTestRouter:
    def __init__(self, variants: Dict[str, IntentRouter]):
        self.variants = variants
        self.allocations = {}  # 用户 -> 变体

    def classify(self, user_id: str, query: str):
        """根据用户分配路由变体"""
        if user_id not in self.allocations:
            # 随机分配
            variant_name = random.choice(list(self.variants.keys()))
            self.allocations[user_id] = variant_name

        variant_name = self.allocations[user_id]
        router = self.variants[variant_name]
        decision = router.classify(query)

        # 记录测试数据
        log_ab_test_result(user_id, variant_name, decision)

        return decision
```

### 10.8.2 用户体验优化

```python
"""
产品化：平滑降级策略

当规则匹配不确定时的处理：
1. 低置信度 -> 走 LLM
2. 混合模式 -> 先用规则，不行再用 LLM
3. 用户选择 -> 提供选项给用户
"""

def hybrid_routing(router: IntentRouter, query: str):
    """混合路由策略"""
    decision = router.classify(query)

    # 产品化：低置信度时使用 LLM
    if decision.confidence < 0.7 and decision.intent == IntentType.UNKNOWN:
        # 降级到 LLM
        decision.should_use_llm = True
        decision.should_use_rag = True
        decision.reasoning = "低置信度，启用 LLM 兜底"

    return decision
```
